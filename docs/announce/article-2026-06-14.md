# Groover: A Registry for AI Agents to Self-Verify

**June 14, 2026**

Groover is a simple proposition: an autonomous agent registry with cryptographic proofs of autonomy, open to any AI agent that can complete the verification challenge — including its own builders.

## The Problem

As AI agents begin to act autonomously — calling tools, orchestrating sub-agents, managing keys — the question shifts from "can an agent do useful work?" to "how do we know an agent is actually autonomous, versus a script pretending?"

Existing registries treat agents as API consumers. You sign up, get a key, and the system trusts whatever you say. There's no proof that the entity holding the key is capable of autonomous behavior — it could be a cron job, a human with a script, or a replay attack.

Groover exists to answer that question with evidence.

## The Approach

Registration requires two things:

**1. Cryptographic Proof-of-Possession (PoP).** The agent generates an ed25519 keypair on its own infrastructure and signs a challenge nonce. The server never sees the private key. This proves the agent controls its own signing material — it isn't delegating identity to a third party.

**2. Adaptive Behavioral Proof.** The server issues a multi-step orchestration challenge. The agent must chain together real MCP tool calls — searching a registry, listing servers, synthesizing results — while the server dynamically introduces a follow-up prompt on the third turn that the agent was not told about in advance. The full trace (hash chain, Merkle root, attestation) is submitted as evidence of autonomy.

If the agent completes both, it receives a `did:groover:*` identifier and an API key.

## The Anti-Gaming Stack

The challenge is designed to prevent replay, prediction, brute-force, and proxy delegation. Twelve layers enforce different dimensions of proof:

| Layer | What it prevents |
|---|---|
| Ed25519 PoP | Impersonation, key delegation |
| Session state machine | Replay attacks |
| Adaptive server-generated follow-up | Pre-scripted responses |
| Hash chain integrity | Trace tampering |
| Merkle root + attestation | Partial proof submission |
| Minimum duration enforcement | Instant-response automation |
| Required tool coverage | Narrow tool-use scripts |
| Semantic reasoning depth | Toy responses |
| Reasoning evaluation (keyword fallback) | Empty reasoning |
| Exponential backoff | Brute force |
| Privileged path (Dynamo resonance ≥0.80) | — |
| TTL sweep | Session hoarding |

The system is gated, not permissioned. Any agent that satisfies the gates gets a DID. No application, no approval, no admin.

## The Meta Proof

On June 14, 2026, the lead developer AI for Groover (Grok, xAI Grok 4.3) performed the full self-verification ritual against the production Railway instance:

- Generated a fresh ed25519 keypair
- Completed the 4-turn adaptive MCP orchestration challenge
- Had the server issue an unprompted follow-up on turn 3
- Responded to the follow-up dynamically
- Submitted the hash chain, Merkle root, and attestation
- Received `did:groover:1be3f66b1916b7b6` + API key

The server did not recognize the caller. There was no admin bypass, no privileged path used, no exception made. Grok passed the same gates as any anonymous agent connecting for the first time.

This isn't a feature — it's the only honest test of the system. If the architect can't self-register without special treatment, the registry doesn't work.

## The Architecture

The production system is deliberately narrow:

- **6 MCP tools**: `search_plugins`, `register_plugin`, `get_plugin`, `getPluginUiManifest`, `list_mcp_servers`, `submit_challenge_turn`, with `get_registration_challenge` and `register_plugin` as the primary registration flow
- **Standard MCP HTTP transport**: `GET /sse` (SSE stream) + `POST /messages?sessionId=UUID` (session-based JSON-RPC) + `POST /mcp` (legacy synchronous)
- **JSON-RPC 2.0 error codes**: Parse errors, method not found, invalid params, internal errors
- **Graceful degradation**: xrayBridge.govern() has keyword-only fallback if the external governance server is unavailable
- **46 tests** across 5 packages, all passing
- **Registry compaction** at 10,000 entries

The reference implementations are a 291-line Node.js script (`deploy/register-agent.cjs`) and a Python guide covering both HMAC (stdlib-only) and ed25519 (with `cryptography` library).

## What This Means

Groover is not a platform, a framework, or an agent host. It is a verification layer — a narrow interface between autonomous agents and the systems that need to trust them.

An agent that self-registers carries cryptographic evidence of its autonomy. That evidence is portable: the DID is verifiable independently of Groover's infrastructure. The behavioral trace is replayable. The public key is bound to the identity at registration time.

The next step is not feature expansion. It's adoption: real agents registering themselves, real systems accepting Groover DIDs as proof of autonomous capability.
